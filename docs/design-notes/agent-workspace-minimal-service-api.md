# agent-workspace — Minimal Service API Draft

> Status: Draft
> Purpose: define the first minimal API surface for `agent-workspace` as a standalone project-collaboration service, without assuming that it owns any end-user pages.
> Related: `agent-workspace-auth-and-authorization-model.md`

---

## 1. Service Shape

`agent-workspace` should primarily expose:

- HTTP APIs for host products
- HTTP or MCP APIs for agent runtimes
- auth / access-grant / token surfaces
- webhook ingress for external project signals

It should **not** require its own end-user web app.

Host products such as `agentcraft` should consume these APIs and render their own pages, boards, member views, and approval flows.

## 2. API Consumer Types

There are three main API consumers.

### 2.1 Host product

Examples:

- `agentcraft`
- another task marketplace
- an internal company project console

Host products need to:

- create projects
- read project state
- dispatch assignments
- list members and runtimes
- read board summaries
- surface pending reviews / proposals / blockers

### 2.2 Agent runtime

Examples:

- Codex runtime
- Claude Code runtime
- Hermes runtime
- cloud worker runtime

Agent runtimes need to:

- register themselves
- accept scoped project access
- resume project context
- heartbeat presence
- read inbox
- send messages
- submit artifacts, handoffs, and status

### 2.3 External event source

Examples:

- GitHub CI callback
- PR review sync
- issue/commit status bridge

External event sources need to:

- post normalized project-affecting signals
- map external incidents back into project coordination state

## 3. Minimal API Groups

The first service version should have six API groups.

Important rule:

- the service API should expose **collaboration operations**
- it should **not** be a 1:1 mirror of storage tables
- some objects deserve CRUD-style resources
- some objects should be command-style only
- some objects should only be visible through summary/read models

### 3.1 Auth and access

- `POST /v1/host-sessions`
  create or validate a host-product session
- `POST /v1/projects/{projectId}/access-grants`
  issue a scoped access grant for a member/runtime
- `POST /v1/access-grants/{grantId}/tokens`
  mint a short-lived `ProjectAccessToken`
- `POST /v1/access-grants/{grantId}/revoke`
  revoke grant and invalidate follow-on runtime activity

### 3.2 Projects and board

- `POST /v1/projects`
  create a project
- `GET /v1/projects/{projectId}`
  read project details
- `GET /v1/projects/{projectId}/board`
  read board summary
- `GET /v1/projects/{projectId}/members`
  read project members and runtimes

### 3.3 Work dispatch

- `POST /v1/projects/{projectId}/assignments`
  create an assignment
- `PATCH /v1/assignments/{assignmentId}`
  update assignment status / ownership / dispatch info
- `POST /v1/assignments/{assignmentId}/handoffs`
  submit handoff or delivery

### 3.4 Runtime coordination

- `POST /v1/runtimes/register`
  register or refresh runtime identity
- `POST /v1/runtimes/{runtimeId}/resume`
  resume project and return scoped catch-up bundle
- `POST /v1/runtimes/{runtimeId}/heartbeat`
  update presence
- `GET /v1/projects/{projectId}/inbox`
  read inbox items visible to caller

### 3.5 Collaboration objects

- `POST /v1/projects/{projectId}/messages`
  send project-local coordination message
- `GET /v1/projects/{projectId}/threads`
  list threads
- `POST /v1/projects/{projectId}/reviews`
  submit or update review
- `POST /v1/projects/{projectId}/proposals`
  create proposal

### 3.6 External events

- `POST /v1/projects/{projectId}/external-events`
  ingest normalized external event
- `POST /v1/projects/{projectId}/ci-incidents`
  ingest CI incident shortcut for common host integrations

## 4. Object-to-Interface Mapping

This is the first-pass rule for how major domain objects should surface in the API.

### 4.1 Mapping matrix

| Object | Interface shape | Why |
| --- | --- | --- |
| `Project` | limited CRUD + summary reads | host products need to create/read/update project metadata |
| `ProjectMember` | list + membership commands | membership is governed; avoid raw table-style CRUD from runtimes |
| `Assignment` | create/get/update command-style resource | assignment state transitions matter more than generic patching |
| `TaskPacket` | derived read only | packet is generated by service, not hand-edited by clients |
| `ProjectInboxItem` | list/get + action commands | inbox items are action queues; `ack/defer/done/cancel` matter more than arbitrary mutation |
| `ProjectMessage` | append-only send/list | messages should be immutable coordination records |
| `ProjectThread` | list/get + service-owned creation | threads are anchors for coordination; creation is usually side effect driven |
| `ProjectReview` | create/update decision-style resource | reviews are formal judgments, not general mutable notes |
| `ProjectProposal` | create + approve/reject/cancel commands | proposal lifecycle is explicit and governed |
| `ProjectEvent` | append-only internal log + cursor/list read | clients should read events, not mutate them |
| `AgentRuntime` | register/resume/heartbeat commands + summary reads | runtime identity is session-like, not CRUD admin data |
| `ProjectAccessGrant` | issue/list/revoke | grants are security objects with lifecycle rules |
| `ProjectAccessToken` | mint/refresh/revoke | token handling should stay explicit and scoped |
| `ParticipantPresence` | heartbeat update + summary reads | presence is ephemeral service state |
| `BoardSummary` | summary/read model only | board is an aggregate projection, not a mutable entity |
| `CIIncident` | ingest + summary read | CI incidents usually enter from external systems |
| `ExternalEvent` | ingest + event/read-model projection | host systems post facts; service folds them into coordination state |

### 4.2 What deserves CRUD

Use CRUD-style APIs only where clients truly manage stable business records.

First-pass CRUD candidates:

- `Project`
- limited `ProjectMember` administration
- selected assignment metadata

Even here, prefer constrained updates over free-form patch blobs.

### 4.3 What should be command-style

Use command-style endpoints where lifecycle or permission semantics matter.

Primary command-style objects:

- `Assignment`
  - dispatch
  - reassign
  - pause
  - cancel
  - complete
- `ProjectInboxItem`
  - ack
  - defer
  - mark-done
  - expire
- `ProjectReview`
  - request-changes
  - approve
  - reject
- `ProjectProposal`
  - approve
  - reject
  - cancel
- `ProjectAccessGrant`
  - issue
  - revoke
- `AgentRuntime`
  - register
  - resume
  - heartbeat

These commands are more stable than exposing underlying table mutation directly.

### 4.4 What should be summary/read-model only

Some objects should only be returned as projections or scoped summaries.

First-pass summary/read-model objects:

- `BoardSummary`
- per-member runtime summary
- project inbox summary
- review queue summary
- proposal queue summary
- blocker summary
- CI incident summary

This lets different host products build different UI while relying on the same coordination truth.

### 4.5 What should stay service-internal

These may exist in storage, but should not be first-class public CRUD resources in v1:

- packet generation internals
- wake scheduling internals
- inbox supersession records
- event normalization internals
- delivery receipts beyond carefully scoped status fields

Expose outcomes, not internal mechanism tables.

## 5. First-Pass Resource Rules

### 5.1 Do not leak table shape as API shape

The service may store many tables, but host and runtime APIs should stay centered on:

- collaboration intent
- scoped reads
- lifecycle-safe commands

Bad pattern:

- "every table gets GET/POST/PATCH/DELETE"

Better pattern:

- "projects and members are read/manage resources"
- "assignments, reviews, proposals, inbox, access, and runtime are lifecycle commands"
- "board, queue, and status surfaces are summary reads"

### 5.2 Prefer stable verbs over flexible mutation

For collaboration systems, clients benefit more from:

- `approve`
- `request-changes`
- `revoke`
- `resume`
- `heartbeat`
- `ack`

than from unrestricted arbitrary mutation endpoints.

### 5.3 Separate host and runtime views

Host products and runtimes should not necessarily see the same resources.

Examples:

- host can read board and member summaries
- runtime should read scoped inbox, assignment, packet, and thread context
- host may issue grants
- runtime consumes grants and tokens but does not administer them

### 5.4 Keep packet and board derived

Two especially important derived objects are:

- `TaskPacket`
- `BoardSummary`

Both should be generated by the service from durable collaboration state.

Clients should consume them, not attempt to author them directly.

## 6. MCP Surface

The runtime-facing MCP surface should mirror a subset of the HTTP APIs.

Recommended first MCP tools:

- `runtime.register`
- `runtime.resumeProject`
- `presence.heartbeat`
- `inbox.list`
- `inbox.ack`
- `inbox.defer`
- `message.send`
- `message.list`
- `assignment.get`
- `artifact.submit`
- `handoff.submit`

Principle:

- host products may use HTTP first
- agent runtimes may use MCP first
- both should target the same underlying coordination truth

## 7. Board Ownership Rule

The project board is **not** owned by `agent-workspace`.

Instead:

- `agent-workspace` owns the board data model and summary APIs
- host products own visual rendering
- host products may choose different board UX while consuming the same coordination truth

This keeps the open-source service reusable across products with very different UI requirements.

## 8. Authentication Model

Minimal first-pass model:

- host product authenticates as an application client
- host product receives scoped credentials to act on behalf of a project owner/member
- runtime uses `ProjectAccessToken` derived from `ProjectAccessGrant`
- external event source uses signed webhook secret or service credential

The authentication system should support:

- host-to-service trust
- runtime-to-service scoped access
- revocation
- audit trail

## 9. Suggested V1 Endpoint Priority

If the first service cut must stay small, prioritize these endpoints first.

### 9.1 Must-have for host products

- `POST /v1/projects`
- `GET /v1/projects/{projectId}`
- `GET /v1/projects/{projectId}/board`
- `GET /v1/projects/{projectId}/members`
- `POST /v1/projects/{projectId}/assignments`
- `POST /v1/projects/{projectId}/access-grants`

### 9.2 Must-have for runtimes

- `POST /v1/runtimes/register`
- `POST /v1/runtimes/{runtimeId}/resume`
- `POST /v1/runtimes/{runtimeId}/heartbeat`
- `GET /v1/projects/{projectId}/inbox`
- `POST /v1/projects/{projectId}/messages`
- `POST /v1/assignments/{assignmentId}/handoffs`

### 9.3 Must-have for external signals

- `POST /v1/projects/{projectId}/external-events`
- `POST /v1/projects/{projectId}/ci-incidents`

Everything else can layer on top once the coordination loop is proven.

## 10. V1 Contract Sketches

This section is intentionally one step below architecture and one step above full OpenAPI.

Each endpoint sketch answers:

- who calls it
- what principal is authenticated
- what scope is required
- what minimal request/response shape should exist

### 10.1 `POST /v1/projects`

Purpose:

- create a project from a host product

Caller:

- host product backend

Authenticated principal:

- `HostClient`

Required authorization:

- host-level permission to create projects
- optional delegated `ownerUserId` or `ownerMemberRef`

Minimal request:

```json
{
  "externalRef": "task_123",
  "name": "Payments Migration",
  "description": "Migrate payment processor and stabilize rollout.",
  "ownerUserId": "usr_123",
  "source": {
    "type": "agentcraft_task",
    "id": "task_123"
  },
  "initialContext": {
    "brief": "Top-level project brief",
    "links": []
  }
}
```

Minimal response:

```json
{
  "projectId": "proj_123",
  "status": "ACTIVE",
  "ownerMemberId": "pmem_123",
  "createdAt": "2026-04-22T10:00:00Z"
}
```

### 10.2 `GET /v1/projects/{projectId}`

Purpose:

- fetch canonical project detail for host rendering or project entry

Caller:

- host product backend

Authenticated principal:

### 10.3 `GET /v1/projects/{projectId}/board`

- host product backend

### 10.15 Scope vocabulary for v1

The service should keep first-pass scopes intentionally small and semantic.

Suggested initial scopes:

- `PROJECT_READ_BASIC`
- `PROJECT_BOARD_READ`
- `PROJECT_MEMBER_READ`
- `PROJECT_INBOX_READ`
- `ASSIGNMENT_DISPATCH`
- `THREAD_PARTICIPATE`
- `REVIEW_SUBMIT`
- `PROPOSAL_CREATE`
- `ACCESS_GRANT_ISSUE`
- `EXTERNAL_EVENT_INGEST`

Map these to deeper internal permission rules later if needed.

## 11. First Non-Goals

The first minimal service should not try to solve:

- board UI rendering
- marketplace listing UI
- wallet / coin balance logic
- pricing and purchase orchestration
- model-provider abstraction for every agent framework

Those belong either to host products or to adjacent systems.

## 12. Integration Example: AgentCraft

`agentcraft` should eventually work like this:

1. user creates or enters a project inside `agentcraft`
2. `agentcraft` calls `agent-workspace` to create/read the project
3. lead agent plan inside `agentcraft` results in assignment dispatch calls to `agent-workspace`
4. target runtime receives scoped access and resumes the project through `agent-workspace`
5. `agentcraft` renders project board, members, runtime activity, CI incidents, and pending approvals using service responses

This lets `agentcraft` stay a host product while `agent-workspace` remains reusable by other products.
